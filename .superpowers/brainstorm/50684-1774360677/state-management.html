State Management Architecture

How data flows through the scheduler feature

Server State — TanStack React Query

All data fetched from the API. Cached, refetchable, invalidated on mutations.

useAssignments(semesterId, filters)

Fetches all assignments for the grid. Returns flat array, indexed client-side by [deptId][weekNum].

useDepartments()

Fetches department list for Y-axis. Cached long-term (rarely changes).

useConstraints(semesterId)

Fetches all constraints (iron, date, department, holidays). Used for client-side validation + blocked cells.

useSemesterWeeks(semesterId)

Fetches week definitions (number, startDate, endDate) for X-axis headers.

Mutations (with optimistic updates)

useMoveAssignment()

Drag & drop move. Optimistic update → API call → rollback on failure + toast.

useCreateAssignment()

Manual dialog submit. Invalidates assignments query on success.

useUpdateAssignment()

Edit dialog save. Invalidates assignments query on success.

Client State — Zustand Store

UI-only state. Filter selections, dialog open/close, active drag state.

useSchedulerStore 
{
  // Filters
  selectedUniversities: string[]
  selectedShift: 'all' | 'morning' | 'evening'
  selectedYear: number | null

  // Dialog state
  activeDialog: null | 'create' | 'import' | 'edit'
  editingAssignmentId: string | null

  // Drag state
  activeDragId: string | null

  // Actions
  setFilter(key, value)
  openDialog(type, assignmentId?)
  closeDialog()
}

Why Zustand (not React Query)?

  • Filters are local UI state — not from server
  • Dialog open/close is ephemeral
  • Drag state changes rapidly during interaction
  • Zustand is already used in the project (themeStore)
  • Filters could be URL-synced later via middleware

Derived State — useMemo Hooks

Computed from server + client state. No separate store needed.

useGridData()

Combines assignments + filters → 2D map: Map<deptId, Map<weekNum, Assignment[]>>. Filters are applied here.

useBlockedCells()

Combines constraints + holidays → Map<cellKey, BlockReason>. Used for visual blocking + drop validation.

Drag & Drop Flow

1. User starts drag → DndContext.onDragStart → store.setActiveDragId()
2. Card follows cursor → DragOverlay renders ghost card
3. Hover over cell → GridCell highlights (green = valid, red = blocked)
4. User drops card → DndContext.onDragEnd fires
5. Client validates → check useBlockedCells + capacity
6a. ❌ Invalid → snap back + toast error with reason
6b. ✅ Valid → optimistic update (move card in cache)
7. API call: PATCH /assignments/:id/move
8a. ❌ Server rejects → rollback optimistic update + toast
8b. ✅ Server confirms → invalidate query (re-sync)
